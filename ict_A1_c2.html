  • 數據有效性檢驗 vs 數據驗證：兩者概念不同，前者由電腦自動執行規則，後者驗證輸入是否與原始數據一致（如雙重輸入、核對）

數據分級

Data Hierarchy

數據通常以分級形式儲存。在數據分級中，每個較高層的數據都是較低層項目的集合。

📦 數據庫 Database 例：學校整個數據庫系統
📁 檔案／表 File / Table 例：學生資料表、課程表
📄 記錄／列 Record / Row 例：某一學生的所有資料
🔤 欄位 Field 例：學生姓名、性別、班別
層級（低→高）名稱說明例子
最低層 欄位 (Field) 儲存單一屬性的數據單位 學生姓名、性別
↑ 記錄 (Record / Row) 集合某實體的所有欄位 學生 ID、姓名、性別、班別、學號
↑ 檔案 / 資料表 (File / Table) 集合結構相同的記錄 儲存了 100 名學生資料的數據表
最高層 數據庫 (Database) 集合多個相關數據表 學校整個數據庫（含學生表、課程表等）
⚠️ DSE 常見錯誤：排序題

DSE 曾考「由最低層至最高層排列」：正確順序為
欄位 → 記錄 → 檔案（表）→ 數據庫
注意：「檔案」與「記錄」的順序最易混淆！

數據庫管理系統（DBMS）

在管理大量數據時，數據庫通常比試算表更好。數據庫管理系統 (Database Management System, DBMS) 透過圖形用戶界面 (GUI) 讓我們有效地組織、儲存和提取數據。

💡 試算表 vs 數據庫
比較試算表 (Spreadsheet)數據庫 (Database)
適合規模小量數據大量數據
數據關聯難以建立多表關聯可建立表間關聯（如外鍵）
查詢能力函數（VLOOKUP 等）SQL 語言
多用戶有限支援多用戶同時存取

數據控制 — 數據有效性檢驗

Data Control — Data Validation
⭐ DSE 高頻考點

數據有效性檢驗的各種類型是 DSE 必考題材，每年至少 2–3 道題涉及此內容。必須能夠：(1) 識別各種類型，(2) 根據場景判斷應使用哪種類型，(3) 區分「有效性檢驗」與「數據驗證」的分別。

無用輸入，無用輸出（GIGO）

當不可靠或不協調的數據輸入到數據分析中時，便不能得出有用的結果，這稱為 Garbage In, Garbage Out (GIGO)。因此，我們必須對數據加以控制。

常見輸入錯誤類型

錯誤類型說明例子控制方法
轉錄錯誤 從一個媒體抄到另一個媒體時出錯 OCR 將大楷「I」識別為「l」或「1」 拼寫檢查
打字錯誤 鍵盤輸入時按錯鍵 將「治」打成「冶」 拼寫檢查
換位錯誤 數據位置不正確（順序顛倒） 將「$15」打成「$51」 核對數據
不存在的數據 提供了不存在的值 輸入不存在的電郵地址 格式檢查、驗證回覆

數據有效性檢驗的八種類型

數據有效性檢驗 (Data Validation) 是數據控制的一部分，由電腦系統自動執行。它不能保證數據絕對正確，但確保數據合理並符合某些規則。

①
完整性檢查
Presence Check
確保必填欄位中有數據，不可留空。
申請人必須提供姓名。
②
固定值檢查
Restricted Value Check
確保數據只能來自一系列特定的值。
申請人的性別必須是「M」或「F」。
③
類型檢查
Type Check
確保數據必須為特定數據類型。
申請人的年齡必須是數字（不能是文字）。
④
範圍檢查
Range Check
確保數據必須位於指定範圍內。
申請人的年齡必須大於 12 歲。
⑤
長度檢查
Length Check
確保數據的字符數必須在指定範圍內。
申請人必須提供 8 個位數的電話號碼。
⑥
格式檢查
Format Check
確保數據必須採用特定格式。
電郵地址必須包含「@」符號。
⑦
一致性檢查
Consistency Check
確保多個欄位之間的數據具有合理關係。
結束日期必須在開始日期之後。
⑧
唯一性檢查
Uniqueness Check
確保欄位的數據必須是獨一無二的。
每位學生只能申請一次，學生 ID 不應重複。

用戶界面工具輔助有效性檢驗

UI 工具輔助的檢驗類型說明
下拉式選單固定值檢查只可選取預設選項之一
單選按鈕固定值檢查只可選取一個特定值
複選框固定值檢查可選取多個預設值
日期選擇器格式檢查防止輸入「32/08/2025」等無效日期

數據類型（Data Types）

為方便組織和分析數據，同一欄位的數據應屬同一數據類型。

數據類型說明數值例子用途例子
整數 (Integer)可以是正數、負數或零，無小數位-29, 0, 100學生分數、年齡
實數 (Real)可以有小數點和小數位3.14, -2.5身高、體重
字串/文字 (String)由字符組成（英文、數字、中文、符號）"陳大文", "1A"姓名、班別
布爾 (Boolean)只能是 TRUE（真）或 FALSE（假）TRUE, FALSE是否合格、是否在世

檢查數位

Check Digit
⭐ DSE 必考計算題

檢查數位（尤其香港身份證 HKID 算法）是 DSE 每隔幾年必考的計算型題目，正確率偏低。必須掌握完整計算步驟。

檢查數位 (Check Digit) 是一種數據有效性檢驗方式，常用於數字數據，如：身份證號碼、書號（ISBN）、信用卡號碼。它由特定算法生成，並附加在數據末尾。若輸入數據的檢查數位與算法計算結果不匹配，則該數據無效。

🆔 香港身份證號碼（HKID）檢查數位 — 逐步詳解

  1. 格式化：忽略括號。若身份證只有一個英文字母，在最前面補一個空格，使總長度統一為 9 個字符。
    一個字母： P246246(A) → 補空格後：[空格]P246246A
    兩個字母：WX135135(2) → WX135135 2
  2. 字母轉數字：A=10, B=11, C=12, … Z=35，空格=36。
    空格→36, P→25, W→32, X→33
  3. 乘以比重：從左到右，9 個數字（含轉換後的字母）分別乘以比重：
    位置第1位第2位第3位第4位第5位第6位第7位第8位第9位（檢查數位）
    比重987654321
    將各乘積加總，得到加權和 (Weighted Sum)。
  4. 整除檢驗：加權和 ÷ 11，若餘數為 0，則號碼有效。
    若要求計算檢查數位：加權和（不含檢查數位）÷ 11 = 餘數 r
    → 檢查數位 = 11 − r（若結果為 10，則檢查數位為 A；若為 11，則為 0）
✅ 計算例子：驗證 P246246(A)

空格→36，P→25；數字：2, 4, 6, 2, 4, 6；檢查數位：A=10

加權和 = 36×9 + 25×8 + 2×7 + 4×6 + 6×5 + 2×4 + 4×3 + 6×2 + 10×1
= 324 + 200 + 14 + 24 + 30 + 8 + 12 + 12 + 10 = 634

… 實際課堂計算以教材原始數字為準（OCR 可能存在輕微偏差）

餘數為 0 → 號碼有效 ✓

奇偶檢測

Parity Check

數據以二進制代碼儲存在電腦中。為檢測數據傳輸過程中的錯誤，在二進制代碼末尾附加一個奇偶檢驗位 (Parity Bit)，以確保代碼中 1 的總數是偶數或奇數。

偶數奇偶檢測 (Even Parity)

確保二進制代碼中 1 的總數為偶數。

  • 原始：0000 001（一個 1）→ 奇偶位 = 1 → 0000 0011（兩個 1，偶數）
  • 原始：1010 101（四個 1）→ 奇偶位 = 0 → 1010 1010（四個 1，偶數）

奇數奇偶檢測 (Odd Parity)

確保二進制代碼中 1 的總數為奇數。

  • 原始：0000 001（一個 1）→ 奇偶位 = 0 → 0000 0010（仍一個 1，奇數）
  • 原始：1010 101（四個 1）→ 奇偶位 = 1 → 1010 1011（五個 1，奇數）
⚠️ 奇偶檢測的限制
  • 只能檢測奇數個位元出錯的情況（如 1 個位元翻轉）
  • 若偶數個位元同時翻轉（如 2 個），1 的總數不變，錯誤無法被檢測
  • 通過奇偶檢測不代表數據傳輸一定沒有錯誤
💡 DSE 解題技巧：當題目不說明使用哪種奇偶檢測

題目給出多組代碼，說明「只有一組通過」→ 分別數各組中 1 的數目：

  • 若只有一組 1 的數目為奇數 → 應用的是奇數奇偶檢測
  • 若只有一組 1 的數目為偶數 → 應用的是偶數奇偶檢測

數據驗證

Data Verification
⚠️ 必須區分！有效性檢驗 vs 數據驗證
比較數據有效性檢驗 (Validation)數據驗證 (Verification)
執行方式由電腦/系統自動執行規則由人手或電腦比對原始數據
目的確保數據合理、符合規則確保輸入數據與原始數據一致
能否保證正確❌ 不能保證絕對正確✅ 可驗證一致性
例子年齡必須 > 12雙重輸入電郵地址並比對

數據驗證的三種方法

方法說明例子
輸入數據兩次
(Double Data Entry)		 由兩位操作員分別輸入相同的數據，比較兩份複本是否相同。 更改電郵地址時，要求用戶輸入新電郵地址兩次，系統比對是否一致。
視覺核查
(Visual Check)		 由同一操作員將輸入的數據與原始資料比較，確認一致。 在電子表單填好後，用戶自行再次對照原始文件確認。
核對數據
(Proofreading)		 由另一位操作員將輸入之數據與原始檔案比對。 將學生成績輸入電子成績單後，打印出來與試卷互相核對。

數據庫的功能

Functions of a Database (DBMS)
⭐ 2025 DSE 考試重點（卷一乙 Q2、Q5）

SQL 語句（INSERT、SELECT、WHERE、ORDER BY）及索引的原理與代價是高頻考點。必須能分辨各 SQL 語句的作用，以及理解索引的利弊。

① 組織（數據庫結構）

良好的數據庫結構可以使不同數據表之間的傳遞更流暢，並減少數據重複。選擇適合的數據類型，使數據更一致，分析更輕鬆。

② 儲存（INSERT / GUI）

可以使用 SQL 語句或 GUI 在數據表中插入新記錄、更改或刪除現有記錄。

INSERT INTO STUDENT (SID, ENAME, SEX, CLASS, CNO) VALUES ("015044", "Peter Chan", "M", "1A", "11");

亦可將 .csv（逗號分隔值）檔案匯入數據庫，在短時間內創建大量記錄，節省手動輸入時間。

③ 提取（SELECT / WHERE / ORDER BY）

使用 SQL 或 GUI 從數據庫中提取特定數據，並可按標準排序。

-- 提取所有男生 SELECT * FROM STUDENT WHERE SEX = "M"; -- 按姓名排序 SELECT * FROM STUDENT WHERE SEX = "M" ORDER BY ENAME; -- 計算記錄數（2025 DSE 考點） SELECT COUNT(*) FROM STUDENT WHERE SEX = "F"; -- 返回：1 行，顯示符合條件的記錄數目
⚠️ 2025 DSE 常考陷阱：COUNT(*) 返回幾行？

SELECT COUNT(*) FROM STUDENT WHERE SEX = "F" 的結果只有1 行，內容是符合條件的記錄數目（例如：4）。很多考生誤以為返回所有女生記錄，需特別留意。

④ 索引（Index）

建立索引後，DBMS 會建立一個索引表（Index Table），僅包含索引欄位和指向相關記錄的指針。搜尋時，DBMS 可對索引表使用二分搜尋法 (Binary Search)，大幅減少比較次數。

💡 索引搜尋過程（Binary Search 示例）

搜尋「Peter Chan」（11 條記錄的索引表）：

  1. 檢查第 6 項（Liliana Sykes，以「L」開頭）→「L」在「P」前 → 排除前 6 項
  2. 檢查第 9 項（Roberta Barrera，以「R」開頭）→「R」在「P」後 → 排除後 3 項
  3. 檢查第 7 項（Nolan Humphrey）→ 不符合 → 只剩 1 項
  4. 最後一項即「Peter Chan」

只需 4 次比較，而非逐行讀取 11 條記錄。

索引的利與弊

說明
✅ 優點大幅加快數據提取/搜尋速度
❌ 代價 1需要額外的儲存空間儲存索引表
❌ 代價 2加入新記錄或修改欄位後，須更新索引表（需時間）
⚠️ 建議只在必要時才建立索引，為太多欄位建立索引並不可取

檔案存取方式

File Access Methods

數據可以儲存在不同的儲存設備中，它們決定了存取數據的方式。常見的兩種存取方式：

📼 順序存取
Sequential Access

  • 從頭開始依次讀寫數據
  • 存取特定數據須依次搜索，速度較慢且難以預測
  • 存取速度視乎目標數據所在位置而定
  • 成本較低
  • 適合：儲存備份數據（按順序寫入）

設備例子：磁帶 (Magnetic Tape) 🎞️

💿 直接存取
Direct Access

  • 每個儲存位置有獨一無二的位址
  • 可直接存取任何位置，無須搜索其他位置
  • 速度更穩定，一般比順序存取快
  • 成本較高
  • 適合：儲存常用數據（需頻繁隨機存取）

設備例子：硬碟 💽、USB 隨身碟 🔌、光碟 💿、SSD

順序存取 vs 直接存取 比較表

比較項目順序存取直接存取
存取速度難以預測，通常較慢穩定，通常較快
成本較低較高
儲存密度較高較低
適用場景儲存備份（按序寫入）儲存常用數據（頻繁讀取）
設備例子磁帶硬碟、SSD、USB、光碟

DSE 2012 1A Q01
Q1. 要以檢查數位來驗證一組香港身份證號碼的有效性，我們應：
(1) 確保號碼開首的字符為一或兩個英文字母。
(2) 確保號碼的加權和可被 11 整除。
(3) 確保號碼不可包含多於 9 個字符（括號除外）。
  • A. 只有 (1)
  • B. 只有 (2)
  • C. 只有 (1) 和 (3)
  • D. 只有 (2) 和 (3) — 答案：B（只有 (2)）
答案：B（只有 (2)）
分析：(1) 是格式檢查（字母開頭）；(3) 是長度檢查（9 個字符）；只有 (2) 是真正利用檢查數位（加權和可被 11 整除）的有效性檢驗方式。
DSE 2013 1A Q09
Q2. 某數據傳輸系統接收了四組二進制代碼，如只有一組代碼通過了奇偶檢測，它是以下哪一項？
A. 1101 1000   B. 1110 1011   C. 0010 0101   D. 0100 1000
  • A. 1101 1000（四個 1）
  • B. 1110 1011（六個 1）
  • C. 0010 0101（三個 1）✓
  • D. 0100 1000（兩個 1）
答案：C
各選項 1 的數目：A=4，B=6，C=3，D=2。A、B、D 均為偶數個 1，C 為奇數個 1。若應用偶數奇偶檢測，則 A、B、D 皆通過（三組），不符。若應用奇數奇偶檢測，只有 C（三個 1）通過，符合題目「只有一組通過」的條件。
DSE 2015 1A Q22
Q3. 一個外賣速遞網站的客戶意見表中，客戶拉動滑塊（1–5）評分。若改為文字方塊，應運用哪些有效性檢驗？
(1) 唯一性檢查   (2) 類型檢查（必須輸入 0–5 的數字）  (3) 範圍檢查（值在 0–5 之間）
  • A. 只有 (1) 和 (2)
  • B. 只有 (2) 和 (3) ✓
  • C. 只有 (3) 和 (1)
  • D. (1)、(2) 和 (3)
答案：B
評分值必須是數字（類型檢查），且必須在 1–5 之間（範圍檢查）。唯一性檢查不適用，因為多位客戶可給相同評分。
DSE 2017 Q05
Q4. 老師想將 1A 班學生資料儲存在數據表中，欄位 CNO（班號）的數據類型為數字。以下哪些有效性檢驗適合應用於 CNO 欄位？
(1) 唯一性檢查   (2) 類型檢查   (3) 完整性檢查
  • A. 只有 (2)
  • B. 只有 (1) 和 (2)
  • C. 只有 (2) 和 (3)
  • D. (1)、(2) 和 (3) ✓
答案：D
(1) 同一班所有學生的班號相同，不同班的班號不應重複 → 唯一性；(2) CNO 是數字類型 → 類型檢查；(3) 每個學生必須有班號，不可留空 → 完整性檢查。三者皆適用。
DSE 2009 1A Q11
Q5. 阿發更改帳戶電郵地址時，網上表單出現「無效的電郵地址」和「請輸入相同的電郵地址」的錯誤訊息。此表單應用了哪些數據控制方式？
(1) 完整性檢查   (2) 格式檢查   (3) 輸入數據兩次作驗證
  • A. 只有 (1) 和 (2)
  • B. 只有 (1) 和 (3)
  • C. 只有 (2) 和 (3) ✓
  • D. (1)、(2) 和 (3)
答案：C
「無效的電郵地址」→ 格式檢查（必須包含 @）；「請輸入相同的電郵地址」→ 雙重數據輸入（輸入兩次作比對驗證）。「舊電郵地址」欄不可留空 → 完整性，但題目沒有錯誤訊息指向此，故只有 (2) 和 (3)。
2025 DSE 卷一乙 Q2
Q6. 執行以下 SQL 語句後，返回結果有多少行？
SELECT COUNT(*) FROM STUDENT WHERE SEX = "F"
（假設 STUDENT 表中共有 5 名女生）
  • A. 0 行
  • B. 1 行（內容為數字 5）✓
  • C. 5 行（顯示每位女生的記錄）
  • D. 顯示所有欄位標題
答案：B
SELECT COUNT(*) 是聚合函數，返回的結果只有 1 行，內容是符合 WHERE 條件的記錄總數（如 5）。這是 2025 DSE 卷一乙 Q2 的核心考點，正確率不高。

Chapter Summary
📌 DSE 必記要點
  • 數據分級由低至高：欄位 → 記錄 → 檔案（表）→ 數據庫
  • GIGO 原則：Garbage In, Garbage Out——輸入垃圾，輸出垃圾
  • 8 種有效性檢驗：完整性、固定值、類型、範圍、長度、格式、一致性、唯一性
  • 有效性檢驗 ≠ 數據驗證：前者電腦自動執行，後者比對原始數據
  • HKID 檢查數位：加權和 ÷ 11，餘數為 0 才有效
  • 奇偶檢測：只能檢測奇數個位元出錯；「只有一組通過」→ 判斷奇/偶數檢測類型
  • SELECT COUNT(*) 只返回 1 行（記錄數）
  • 索引：加快搜尋，但需要儲存空間，更新記錄時需更新索引表
  • 直接存取：速度快且穩定，適合頻繁隨機存取（如硬碟）
  • 順序存取：速度慢但成本低，適合備份（如磁帶）
